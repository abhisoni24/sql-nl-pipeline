## More Updates:

### 1. Structural Validation & Baseline Testing

A core baseline test suite was established to ensure that Natural Language (NL) prompts are clean, human-readable, and devoid of technical "leaks".

- **NL Prompt Integrity:** 62 named checks were developed, facilitating 51,786 individual validations.
- **Leak Prevention:** The suite prevents raw SQL keywords (e.g., `FROM`, `WHERE`) or Python technical representations (e.g., `[None]`) from appearing in NL output.
- **Fidelity Across Complexities:** Validations span Simple, Join, Advanced (Subqueries/Exists), and DML operations (Insert/Update/Delete).
- **Current Status:** A **100% Pass Rate** was achieved on the baseline dataset of 3,500 records.

---

### 2. SQL Generation & Contract Validation

To maintain logical consistency, a 37-contract validation pipeline was implemented for generated SQL queries.

- **Multi-Layered Checks:** Every query must pass "All Complexity" checks, such as being parseable SQL with valid schema references, before type-specific validation.
- **Logical Constraints:** \* **Join:** The pipeline validates Foreign Key (FK) pairs and legal join types.
- **DML:** The system ensures `id` columns are not updated and `WHERE` clauses are present for `DELETE` operations to prevent accidental data loss.

---

### 3. Systematic Perturbation Testing

Thirteen specialized Python scripts were created to verify "perturbed" NL prompts, which simulate messy or idiomatic input from human users.

- **Scale:** Over **449,812 total checks** were performed with zero failures following a rigorous calibration phase.
- **Categories Tested:**
- **Linguistic Noise:** Includes typos, urgency qualifiers, and verbosity variations.
- **Structural Noise:** Includes mixed SQL/NL, incomplete join specifications, and anchored pronoun references.

- **Methodology:** The project shifted from "snapshot" testing to **Generalized Property Tests**. The logic (e.g., verifying if a prompt increases in length after adding verbosity) remains independent of the dataset, allowing for portability to other domains like Healthcare or E-commerce.

---

### 4. Analysis of LLM Perturbation Failures

Running the systematic tests against perturbations generated by an LLM revealed significant "noise-handling" failures, providing critical data for security research.

While the internal generator maintained 100% accuracy, the LLM-generated perturbations failed **1,793 checks**. Notable failure patterns include:

- **Union Connector Loss:** In categories such as `anchored_pronoun_references` and `urgency_qualifiers`, the LLM consistently omitted the connector phrase in `UNION` queries, which breaks the logical structure of the query.
- **Content Hallucination:** In `comment_annotations`, 207 cases failed the `baseline_prefix_intact` check, indicating the LLM altered the original intent rather than merely adding an annotation.
- **DML Intent Degradation:** For `omit_obvious_operation_markers`, 33 cases failed to preserve the "update-action cue," rendering the intent to modify data unrecognizable.

#### Example of LLM Failure:

- **Baseline Requirement:** Preservation of condition values and pronouns.
- **Observed Error:** The LLM failed to include pronoun anchors (e.g., "those," "them"), resulting in a loss of reference for follow relationships.
